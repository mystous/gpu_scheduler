# 부하 스윕 종합 결론 (Philly 111k, 시뮬레이터)

부하 256·512·1024 GPU × 단일·이종 클러스터 × 전 스케줄러. 최신 코드(SoA·CAP없음·큐상대 α).
부하배수 = 워크로드 수요 / 클러스터 용량: 256=3.6×, 512=1.8×, 1024=0.9×.

## 1. sfqa-auto: 부하·이질성 높을수록 이득 증가
q_p50을 FIFO 대비 단축: 1024 단일 0% / 512 50%↓ / 256 54%↓ (이종은 58·50·61%↓).
저부하(1024 단일)에선 큐가 안 쌓여 모든 정책 q_p50≈2초로 동일 — 스케줄러 무의미.

## 2. sfqa-auto만 "빠름 + 공정" 동시 달성
공정성 p1(하위1%, 100=완전공정): sfqa-auto 256단일 57·512단일 54·256이종 54·512이종 50.
- FIFO: p1=100이나 q 최악(가장 느림)
- sjf·themis·las: q median 최소지만 p1=0(이종 256/512) — 완전 starvation, q_max 최악
- sfqa-auto: baseline 중 공정성 최고(FIFO 제외) + q_p50 50~61%↓ → 유일한 균형

## 3. 꼬리(q_max) 보호
512 단일 q_max: sfqa-auto 2.52M < sjf 4.69M·themis 3.71M. 짧은 잡만 밀어주다 긴 잡을
방치하는 baseline과 달리 최악 대기를 억제. (256 극과부하선 자리 절대부족 → 모두 ~11M 수렴)

## 4. 이종에서 baseline이 더 무너짐
타입계수(a100/h100가 큰 잡을 오래 점유 → HOL 심화)로 sjf/las/themis p1이 256·512 모두 0 붕괴.
sfqa-auto는 이종서도 p1 50+ 유지("자리 맞는 굶은 잡 승급"이 막힘 완화).

## 5. SOTA(lucid/sia)
- lucid: q_p50 극소(collocation으로 작은 잡 즉시)지만 q_max 큼(꼬리 방치).
- sia: 이종에서 압도적(512 이종 q_p50=200·q_max 473K) — 탄력 스케줄링의 진가. 단일은 평범(Pollux 수렴).

### 5-1. Sia의 계산 overhead — 명백한 단점
라운드마다 '활성 잡 전체 × config goodput' 평가 → 부하↑일수록 라운드 비용·수 폭증.
| 부하 | sia 계산시간 | 비고 |
|---|---|---|
| 1024 (0.9×) | ~수분 | 활성 잡 적음 |
| 512 이종 (1.8×) | ~100분 | |
| 256 단일 (3.6×) | ~2시간 | engine 정책 ~30초의 200배+ |

- goodput-최적 ILP의 정확성을 얻는 대가가 스케줄링 계산비용(부하에 매우 민감).
- 완화: 실제 K8s는 스케줄 주기(60–300s)가 길고 결정 지연이 JCT(수시간)에 무영향 → 운영 부담 작음.
  단 (a) 대규모 시뮬 wall-clock 부담, (b) 초대규모·고빈도 실시간 스케줄링 병목 가능. 논문도 ILP에 시간제한.
- 대조: SQUAD(SFQA/sfqa-auto)는 단일 승급 O(n)·SoA 벡터화로 부하 무관 ~수십 초 — 경량.

## 한 줄 결론
sfqa-auto는 부하·이질성이 심할수록 FIFO 공정성을 유지하며 median을 50~61% 단축하는 유일한 정책.
baseline은 공정하지만 느리거나(FIFO) / 빠르지만 굶기거나(sjf·themis·las) 한쪽으로 치우치고,
sia는 이종 goodput은 우수하나 부하에 비례하는 계산 overhead가 운영·평가 모두에서 부담이다.

## 검증 주의(baseline 신뢰도)
lucid/sia의 파라미터(TYPE_PERF·util 난수·SS_SPEED)는 논문 측정값이 아닌 설정값이며,
predicted_duration=oracle(완벽 예측)이라 baseline에 유리할 수 있음. 절대값보다 경향으로 해석.
방어 가능한 핵심 주장은 "sfqa-auto vs FIFO(파라미터 없는 정확한 baseline)".
