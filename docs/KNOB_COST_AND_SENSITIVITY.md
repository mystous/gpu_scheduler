## 2. v2로도 남는 임의 상수 — 민감도 스윕 설계

JoCC R1("suitability 0.1, priority halving, 3×servers 임의") + IEEE Cloud R5("왜 3배냐") 대응.

### 2.1 대상 상수와 코드 위치

| 상수 | 현재값 | 위치 | 스윕 범위 |
|---|---|---|---|
| R 페널티 (suitability step) | 0.1 | `gpu_scheduer/job_emulator.cpp:420` `1-(avail-i-1)*0.1` | {0.05, 0.1, 0.15, 0.2, 0.3} |
| P 감쇠 밑 (priority halving) | 2 | `gpu_scheduer/job_emulator.cpp:432` `1/pow(2,j)` | {1.5, 2, 3, 4} + 선형 1/(j+1) |
| 큐 prefix 배수 (n=k×서버수) | 3 | 논문 Eq.1 정의(코드: 큐 스캔 범위) | {1, 2, 3, 5, 무제한} |
| τ (BSLD floor, v2 전용) | 10s | `ADAPTIVE_SFQA_DESIGN.md` §2 | {1, 5, 10, 30, 60}s |
| 참고: 기본값 출처 | | `gpu_scheduer/enum_definition.h:3-12` `global_const` | |

### 2.2 실행 방법 (C++ 시뮬레이터 — 인프라 재사용)

1. **상수의 파라미터 승격**: `global_const`의 해당 값을 `scheduler_option` 필드로 옮기고
   `config.set` 7번째 줄 이후에 선택적 라인 추가(`r_penalty`, `p_base`, `prefix_mult`).
   **하위호환**: 라인이 없으면 기존 기본값 — 기존 config.set 전부 그대로 동작.
2. 1-D 스윕(다른 상수는 기본값 고정): 5+5+5 = 15 runs × 트레이스 3종 = 45 runs — 시뮬로 1시간 내.
   (전체 교차 그리드는 불필요 — 목적이 최적화가 아니라 **robustness 입증**이므로 1-D면 충분.)
3. τ 스윕은 K8s 실측(sfqa-auto run 5회) 또는 시뮬 v2 역포팅 후 — 후순위.

### 2.3 판정 기준과 논문 서술

- 각 상수에 대해 **민감도 곡선**(상수값 vs max 큐잉·평균 JCT·할당률) 산출.
- **robust 판정**: 스윕 전 구간에서 핵심 지표 변화 < 5% → "결과는 이 상수에 둔감하며
  관례값을 사용한다"로 서술 — '임의'라는 공격을 '둔감 입증'으로 전환.
- 민감 구간 발견 시: 해당 상수를 v2 방식으로 유도하거나(예: R 페널티를 단편화 비용으로 정의)
  민감 구간을 명시하고 권장 범위 제시.

### 2.5 결과 (2026-06-04)

| 상수 | 스윕 | makespan Δ | 판정 |
|---|---|---|---|
| R 페널티 0.05–0.3 | 5점 | −0.8% ~ +2.0% | **robust** — 0.1은 둔감 구간 |
| P 밑 1.5–4 | 4점 | +0.3% ~ +1.8% | **robust** — 2는 둔감 구간 |
| prefix 배수 m | 1,2,3,5,10 | m=1: **+8.9%**(alloc −6.3pp) / m≥2: ≤1.4% | **m=1만 민감** |

**논문 서술**: "재정렬 창(n=m×서버수)은 m=1이면 SFQA가 볼 수 있는 후보가 부족해 makespan
+8.9% 저하 — 창의 존재 자체가 필요함을 보인다. m≥2부터는 결과가 둔감하며(≤1.4%),
기본값 3은 둔감 구간의 안전한 중앙값이다. R 감쇠·우선순위 밑은 전 스윕 구간에서 5% 미만
변화로, 결과는 이 관례값들에 둔감하다." — '임의 상수' 지적(JoCC R1, IEEE Cloud R5)을
'둔감 입증 + 필요조건 식별'로 전환.
